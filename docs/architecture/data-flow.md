# Data Flow

How a request moves through the system, and where state lives. Implementation
source of truth: `src/App.tsx`, `src/hooks/useUserStore.ts`,
`functions/api/[[path]].js`, `shared/db/`. This page describes the
non-obvious shape.

## Auth lifecycle

1. SPA loads `VITE_GOOGLE_CLIENT_ID` and renders the Google One Tap prompt.
2. On credential selection, the SPA POSTs the Google credential to
   `/api/auth/google`.
3. The Pages Function verifies the credential with Google, upserts a `users`
   row, and sets an httpOnly JWT cookie (`dsa_prep_auth`, 30-day max-age)
   signed with `JWT_SECRET`.
4. Subsequent `/api/*` calls carry the cookie; `requireAuth` middleware
   verifies the JWT and resolves `user_id`.

There is no OAuth redirect flow. Guest mode (no cookie) keeps working with
localStorage-only state.

## Guest vs authenticated state

| Data | Guest | Signed-in |
| --- | --- | --- |
| Artifacts, drills, projects, notes | localStorage | Turso (`user_artifacts`, `user_drills`, `user_projects`, `user_learning_notes`) |
| Concept mastery (FSRS) | Not tracked — requires sign-in | Turso (`concept_mastery`) |
| Chat history, imported problems | localStorage | Turso (`user_chats`, `user_imported_problems`) |
| Progress, activity, ELO | localStorage | Turso (`user_progress`, `activity_log`, `user_elo_state`) |

**Merge on sign-in:** localStorage state is merged into the DB when the user
signs in (`useUserStore`). This is one-way: local → DB. The DB is authoritative
after sign-in; localStorage remains as an offline fallback.

## Static content vs user state

Static content (concepts, roadmaps, drills, artifacts, projects,
review-questions, learning library, learning sources) is JSON in `src/data/`
loaded via `src/data/learning-os.ts`. It is baked into the SPA bundle at build
time. It is **not** fetched from the API at runtime.

The learning-library markdown under `src/data/library/` is generated by
`scripts/fetch-library.mjs` from upstream GitHub repos (see
[`../development/content-pipelines.md`](../development/content-pipelines.md)).
The in-product roadmap markdown under `docs/learning/` is Vite-globbed raw
into `src/pages/LearningDoc.tsx`.

## AI request flow

1. Client calls `useAI` with `aiConfig: {endpointUrl, apiKey, model}` (or
   omits it to use server env fallbacks).
2. `POST /api/chat` (auth required) streams the request.
3. **Prod:** Pages Function forwards to the configured provider via the Vercel
   AI SDK. **Dev:** `vite-plugin-local-ai.js` streams the logged-in
   `claude`/`codex`/`gemini` CLI over SSE — no API keys.
4. The Socratic system prompt forbids direct solutions (see
   [`decisions/0005-socratic-no-solutions.md`](https://github.com/Significant-Hobbies/swe-interview-prep/blob/main/docs/architecture/decisions/0005-socratic-no-solutions.md)).

## Feynman Gate → FSRS loop

1. User solves a drill → skippable explain-back nudge appears.
2. User explains in plain English → AI grades 0–100 + returns gaps.
3. Grade maps onto per-concept FSRS ratings (`feynmanRating`); gaps trigger
   `again`/`hard`.
4. `concept_mastery` is updated; mastery decays over time per the confidence
   formula `(1 + elapsed/(9×stability))^-1`.
5. BuildLab surfaces a "next weakest concept" card → the loop closes:
   drill → explain → mastery update → next weakest concept.

## Go execution flow

1. Client compiles Go in the browser via the WASM interpreter in
   `public/wasm/` (binary hosted on R2 in prod, see `.cfpagesignore`).
2. `/api/go-run` (auth required) proxies to go.dev for execution.
3. Result streams back to the Monaco panel.

## DB initialization

`initDatabase()` runs `CREATE TABLE IF NOT EXISTS …` for every table on first
cold start of the Pages Function. There is no migration runner — schema
changes must be **additive only** and backwards-compatible across deploys.
See `shared/db/schema.mjs` and the parallel init in
`functions/api/[[path]].js` (kept in sync by hand).
